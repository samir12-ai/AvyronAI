This indicates:

The system prevents generic errors, but does not enforce decisive selection of a sharp territory

💡 Proposed Addition — Positioning Compression Layer

This is not a new system layer, but an extension inside the existing Positioning engine.

Objective:

Force the system to:
	•	select one dominant territory
	•	express it as a specific operational/system failure
	•	reject broad or cross-domain-safe outputs

⸻

🧠 Proposed Behavior

1. Selection Constraint (Prompt-Level)

Inside the existing Positioning prompt (RULES block):
	•	Force selection of ONE primary territory only
	•	Reject multiple moderate territories

Example instruction:
	•	“Select only ONE territory. If multiple are similar, reject and reconstruct a sharper one.”

⸻

2. Specificity Pressure (Scoring-Level)

Extend computeSpecificityScore() with:
	•	Penalty if territory applies across multiple industries
	•	Penalty if no system-level noun exists (tool, system, process, framework, etc.)
	•	Penalty if no operational failure is defined

⸻

3. Domain Compression (Prompt-Level)

Require:
	•	Translation → already implemented
	•	Compression → new requirement

Meaning:
	•	Move from emotional description → root cause system failure
