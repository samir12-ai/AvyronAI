Hi Replit Team,

I’d like to align on the model upgrade direction, but based specifically on how our system is actually structured in code — not just a blanket upgrade.

After reviewing the current architecture (#11–#17), I strongly believe we should not upgrade all calls uniformly, but instead adopt a selective (hybrid) model strategy aligned with how different parts of the system operate.

⸻

🧠 Proposed Approach (Architecture-Aligned)

We split model usage into two tiers:

1. Heavy Reasoning / Decision Layers (Upgrade)
These are the parts of the system where correctness and multi-factor reasoning matter most:
	•	Plan synthesis (plan-synthesis.ts)
	•	Agent orchestration decisions (controller loop)
	•	Adaptive rhythm reasoning
	•	Memory mutation decisions
	•	Exploration allocation logic

👉 These should use the strongest reasoning-capable model available

⸻

2. High-Volume / Utility Layers (Keep Lightweight)
These are frequent, structured, or deterministic tasks:
	•	Data extraction / normalization
	•	Formatting / summarization
	•	Tool routing / simple responses
	•	Signal processing (non-decision logic)

👉 These should remain on mini / lightweight models for cost and speed efficiency

⸻

❗ What We Want to Avoid
	•	Upgrading all calls to a heavy model (high cost, low marginal gain)
	•	Keeping everything on mini models (hurts reasoning quality where it matters)
	•	Mixing models without clear responsibility boundaries

⸻

	•	Lightweight models → where volume and speed matter
