⸻

1) Core Rule: Deduplication Must Remain the Final Source of Truth

Even with watermark logic and time-window filtering, the system must always treat post-level deduplication as the final authority.

Requirements:
	•	Use postId / shortcode as the primary deduplication key
	•	Check existing records in ci_competitor_posts before insert
	•	Never allow duplicate posts to reach storage
	•	If external ordering is inconsistent, deduplication must still fully protect the dataset

Watermark and timestamp are optimization layers only. They must not replace hard deduplication.

⸻

2) Explicit Scrape Modes

Please keep the scraper behavior split into two clear modes:

INITIAL

Use when the competitor has no previously stored posts.
	•	Fetch 10–12 posts
	•	Build the initial snapshot
	•	Store the first usable watermark after successful insert

INCREMENTAL

Use when the competitor already has stored posts.
	•	Fetch a lighter batch only
	•	Target 4–5 posts maximum
	•	Focus only on recent activity
	•	Add new posts only

This mode decision should remain centralized in the orchestration layer.

⸻

3) Recent Window Must Be Enforced

This is important: refresh scrapes should not work against the full profile history.

Requirements:
	•	Restrict refresh scraping to a configurable recent window
	•	Default: last 7 days
	•	Allow adjustment to 5–7 days if needed
	•	Do not rely on the overall page history during refresh cycles

This prevents unnecessary scraping, reduces cost, and avoids reprocessing older content repeatedly.

⸻

4) Watermark Behavior Must Be Safe

Please add and use lastPostWatermarkTimestamp, but with the following safeguards:
	•	Watermark should be updated only after a successful scrape
	•	Watermark should be derived from the newest valid newly inserted post
	•	If no new posts are inserted, do not advance the watermark artificially
	•	Watermark should act as a soft boundary, not a hard guarantee

This avoids false progression and protects against missed posts.

⸻

5) Stop Logic Must Be Conservative, Not Aggressive

Please do not stop on the first duplicate or the first old post.

Use a safer stop strategy:
	•	Stop only after a consecutive duplicate threshold is reached (for example 3–4 in a row)
	•	Treat posts older than the cutoff/watermark as a strong signal, but not as a single-point stop condition unless the feed ordering is clearly reliable
	•	The logic should avoid missing newer posts that may still appear in unstable or imperfect ordering

This is important for real-world scraper behavior.

⸻

6) Timestamp Handling Must Be Defensive

Please account for incomplete platform data.

Requirements:
	•	If a post has a valid timestamp, apply cutoff logic normally
	•	If a post is missing a timestamp, do not immediately discard it
	•	Allow it to pass through deduplication checks
	•	Only reject it if there is another explicit reason to do so

This avoids losing potentially valid new posts.

⸻

7) Append-Only Storage

Please keep storage behavior strictly append-only.

Requirements:
	•	Never overwrite existing posts
	•	Never replace older records during refresh
	•	Only insert posts that are confirmed new
	•	Existing data must remain untouched

This is required to keep historical integrity and make later analytics reliable.

⸻

8) Existing Guardrails Must Not Break Incremental Mode

Please make sure the current degradation protection logic does not incorrectly flag incremental fetches as failure.

Specifically:
	•	Any logic that compares a small incremental batch against the historical total must be bypassed or adjusted for INCREMENTAL mode
	•	A smaller fetch count in incremental mode is expected behavior, not degradation

This is a critical protection against false negatives.

⸻

9) Database and Concurrency Safety

Please make the implementation resilient beyond the happy path.

Requirements:
	•	Add or enforce a uniqueness guarantee for postId / shortcode at the database level if possible
	•	Protect against duplicate insert attempts in concurrent runs
	•	Ensure watermark updates do not corrupt state if parallel jobs happen
	•	Failed scrapes must not partially advance competitor state

This is important to avoid silent data corruption later.

⸻

10) Output / Logging Expectations

Please include enough logging to make debugging practical.

At minimum, log:
	•	scrape mode used (INITIAL vs INCREMENTAL)
	•	recent window applied
	•	number of fetched posts
	•	number of inserted posts
	•	number of duplicates skipped
	•	number of posts skipped for cutoff
	•	stop reason
	•	watermark before and after scrape

This will help validate the logic quickly in Replit and in production.

⸻

11) Final Desired Behavior

The final behavior should be:
	•	First scrape → collect a full initial batch
	•	Refresh scrape → collect only recent, new posts
	•	No duplicates
	•	No overwrite
	•	No unnecessary full-history refresh
	•	Only successful new data should move the watermark
	•	System must remain safe under imperfect feed ordering and partial failures
