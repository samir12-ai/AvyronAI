I want to align on the best architecture for competitors persistence and reuse, based on the current codebase.

My proposed direction is hybrid, not fully per-account and not fully global:

## Proposed architecture

### 1) Global shared scraped data layer
A backend-only shared pool for raw competitor intelligence data, such as:
- normalized domain / handle
- scraped profile data
- posts / comments / metrics
- last scraped timestamp
- freshness / TTL status
- scrape quality / completeness markers

Goal:
- avoid repeated scraping for the same domain/handle across multiple accounts
- reuse recent valid data when it already exists
- keep this layer completely invisible to end users in the UX

### 2) Per-account ownership / presentation layer
A separate account-scoped layer for:
- which competitors a given account has added
- when they added them
- which ones should appear in that account’s dashboard
- account-specific notes, labels, workflow state, analysis state, etc.

Goal:
- preserve account isolation in the product experience
- avoid mixing user-facing state between different accounts
- let multiple accounts benefit from the same underlying scraped source data without duplicating scraping work

## Expected behavior
When a user adds a competitor:
1. normalize the domain / handle
2. check whether recent valid scraped data already exists in the shared pool
3. if valid shared data exists, use it directly and link it to that account
4. if not, scrape fresh data, save/update the shared pool, then link it to the account
5. never expose “cache”, “database”, “shared pool”, or similar concepts in the UX
