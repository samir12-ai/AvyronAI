Hi,

Thanks for digging into the codebase — this is exactly the level of clarity we needed. The current foundation is strong, and I agree that we don’t need to rebuild anything, just extend it properly.

I’ve reviewed your findings and refined the plan to make it robust, production-safe, and aligned with long-term scalability. Below is the finalized implementation plan:




1) Core Principles (Non-Negotiable)


	•	Post-level deduplication is the source of truth 
	◦	Always rely on postId / shortcode to prevent duplicates
	◦	Timestamp and watermark are supporting mechanisms, not replacements
	•	
	•	Append-only data model 
	◦	Never overwrite or replace existing posts
	◦	Only insert new posts
	•	
	•	Deterministic + safe behavior 
	◦	No assumptions about ordering reliability from external platforms
	◦	System must be resilient to partial failures and inconsistent feeds
	•	





2) Scrape Modes


Introduce explicit scrape modes:

	•	INITIAL 
	◦	Triggered when no posts exist for a competitor
	◦	Fetch: 10–12 posts
	•	
	•	INCREMENTAL 
	◦	Triggered when existing posts are present
	◦	Fetch: smaller batch (4–5 posts max)
	◦	Must focus on freshness and efficiency
	•	


This logic should live in fetch-orchestrator.ts.




3) Recent Window Constraint (Critical)


We must enforce a time-based window:

	•	Only fetch posts from the last 5–7 days (configurable)
	•	Do NOT rely on full page/history scraping during refresh
	•	This reduces cost, latency, and duplication risk


Implementation:

	•	Pass windowDays → data-acquisition.ts
	•	Apply cutoff using timestamp





4) Watermark System (Guiding Boundary)


	•	Add lastPostWatermarkTimestamp to ciCompetitors (in shared/schema.ts)
	•	After each successful scrape: 
	◦	Update it with the newest post timestamp fetched
	•	


Usage:

	•	Acts as a soft boundary
	•	Helps reduce unnecessary fetches
	•	Must NOT replace deduplication logic





5) Deduplication (Hard Constraint)


Inside data-acquisition.ts:

	•	Before inserting: 
	◦	Check existing postId / shortcode in ci_competitor_posts
	•	
	•	Skip duplicates immediately


This is the final gate — no duplicate should ever reach storage.




6) Early Stop Logic (Refined)


Do NOT stop on first duplicate.

Instead:

	•	Stop when: 
	◦	Posts are clearly older than watermark
	◦	OR after a consecutive duplicate threshold (e.g., 3–4 in a row)
	•	


This prevents:

	•	Missing new posts due to unstable ordering
	•	Over-fetching unnecessary data





7) Fetch Strategy (Adaptive)


Inside data-acquisition.ts:

	•	INITIAL: 
	◦	Fetch full batch (10–12)
	•	
	•	INCREMENTAL: 
	◦	Apply: 
	▪	windowDays filter
	▪	reduced fetch size (4–5)
	▪	early-stop rules
	◦	
	•	





8) Orchestration Layer


In fetch-orchestrator.ts:

	•	Detect mode: 
	◦	If no posts → INITIAL
	◦	Else → INCREMENTAL
	•	
	•	Pass: 
	◦	scrapeMode
	◦	windowDays
	•	


Avoid duplicating logic across layers — keep orchestration centralized.




9) Refresh System Integration


In refresh-system.ts:

	•	After successful scrape: 
	◦	Update lastPostWatermarkTimestamp
	•	
	•	Ensure: 
	◦	No update happens if scrape fails or returns invalid data
	•	





10) Edge Cases (Important)


Please account for:

	•	Platforms returning unordered posts
	•	Missing or null timestamps
	•	Duplicate IDs across different formats
	•	Partial scraping failures
	•	Empty incremental runs (no new posts found)


System must handle all gracefully without corrupting data.




11) Replit Compatibility


Please structure the implementation so it is:

	•	Modular
	•	Easy to test in isolation
	•	Directly runnable within our Replit setup





Final Summary


	•	First run → full scrape
	•	Subsequent runs → recent-window + incremental updates
	•	Deduplication → always enforced
	•	Watermark → guiding optimization layer
	•	Data → append-only, never overwritten




If everything looks good on your side, we can proceed with implementation.
