## 1. New Source Architecture Must Be Source-Type Aware

Do not train the system to think in terms of “platform names only.”

The system must learn to think in terms of signal types.

Required signal type model:

### A. Instagram Signals
These are primarily content behavior signals.

They should be used for:
- hook extraction
- CTA pattern extraction
- content angle extraction
- audience pain inference from hook framing
- short-form content style
- storytelling pattern detection
- authority / proof / curiosity / challenge framing

### B. Website Signals
These are primarily product and positioning signals.

They should be used for:
- positioning language
- headline extraction
- offer structure
- pricing model
- funnel CTA analysis
- proof structure
- guarantees
- feature hierarchy
- brand promise

### C. Blog Signals
These are primarily education and topic signals.

They should be used for:
- educational themes
- repeated market questions
- long-form authority themes
- topic clusters
- problem framing
- strategic education patterns

This separation is critical.

The engines must not treat all source inputs as interchangeable text.

This is the most important requirement.

Do not update only Market Intelligence and assume the system is done.

All affected engines must be updated to correctly interpret the new signal structure.

At minimum, the following engines must be re-aligned:

- Instagram content behavior

### Differentiation Engine

- Profile card inputs

### Offer Engine

### Funnel Engine
Must learn from:
- website CTA flows

### Awareness Engine

### Persuasion Engine

### Statistical / Integrity Layer

This must be done deeply, not superficially.

## 3. Single Signal Model Must Prevent Cross-Source Conflict

We must avoid a common failure mode:

Instagram says one thing  
Website says another thing  
Profile says a third thing  
Plan becomes inconsistent

To solve this, implement a clean signal hierarchy.

### Required signal model:

#### Layer 1 — Profile Card
This is the business intent layer and must be treated as persistent business truth.

Examples:
- real offer
- real niche
- real target market
- business claim
- business model

#### Layer 2 — Market Intelligence Sources
This is the external reality layer.

It validates, pressure-tests, and contextualizes profile inputs.

#### Layer 3 — Reconciliation Layer
This is where the engines reconcile:

- what the business claims
- what the market shows
- what the competitor landscape suggests

This is extremely important.

Do not let source layers overwrite each other directly.

Instead:
- preserve each source
- score each source
- reconcile through engine logic

This is how we avoid signal contamination and strategic drift.

---

## 4. Proxy-Based Scraping Must Be Treated as Infrastructure, Not a Quick Hack

The proxy layer must be implemented with discipline.

We are using residential proxy infrastructure to support source collection.  
That solves access, but not extraction quality.

The scraping system must be designed around the following:

### A. Safe Page Scope
Only scrape the pages that matter:

For websites:
- homepage
- pricing
- features/services
- about
- blog index / relevant blog pages

Do not crawl excessively.

### B. Structured Extraction
Do not return raw HTML blobs into the engines.

Extract clean structured fields such as:
- headline
- subheadline
- CTA labels
- offer phrases
- pricing anchors
- proof blocks
- testimonial blocks
- topic titles
- content headings

### C. Signal Classification Before Engine Injection
Before passing scraped data to engines, classify it into:
- positioning signal
- offer signal
- content signal
- educational signal
- proof signal
- CTA signal

The engines should consume normalized signals, not chaotic page text.

### D. Retry / Error Handling
If scraping fails:
- mark source unavailable
- do not fake signals
- do not inject empty output as valid source
- downgrade confidence cleanly

### E. Source freshness
Store source snapshots with timestamps and validity labels.

Do not let stale website snapshots continue influencing current plans invisibly.

Do not force every competitor to provide the same links.

We need a flexible competitor source model:

### Minimum required:
At least one of:
- Instagram
- Website

### Recommended:
- Instagram + Website

### Optional:
- Blog

Do not add YouTube.  
Do not add LinkedIn.

Those add complexity without enough value for the current MarketMind architecture.

This rule must be implemented in both:
- frontend competitor input forms
- backend validation logic

## 6. The System Must Know What to Do When a Source Is Missing

We cannot assume every business has:
- Instagram
- Website
- Blog

Therefore, each engine must have source fallback behavior.

Examples:

### If Instagram is missing
Use:
- website headlines
- offer copy
- blog titles if available

### If Website is missing
Use:
- Instagram content DNA
- profile card
- competitor external signals

### If both are missing
Use:
- profile card
- competitor source intelligence
- low-confidence mode

The engines must not collapse or behave unpredictably when a source is missing.

## 11. Avoid Common Scraping / Signal Integration Mistakes

Please actively avoid the following common failures:

- treating all scraped text equally
- mixing strategic product copy with short-form content signals
- letting website signals override the actual business offer
- ignoring missing-source states
- silently using stale data
- allowing one strong source to completely drown out the others
- failing to surface confidence penalties to the UI
- updating backend extraction without updating agent logic
- updating engine logic without updating plan synthesis

These are exactly the types of mistakes that create hidden instability.
