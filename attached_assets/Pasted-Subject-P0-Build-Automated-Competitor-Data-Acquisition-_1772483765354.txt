Subject: P0 — Build Automated Competitor Data Acquisition Layer for MIv3

Replit,

MIv3 is structurally sealed.
The engine is strong.
The bottleneck is data acquisition.

Right now Competitor Intelligence runs in Exploratory Mode because competitors have:
	•	0 posts
	•	0 comments
	•	Missing CTA patterns
	•	Insufficient qualified competitors

This is not an engine weakness.
This is a data layer gap.

We are moving to full automation.

Below is the execution plan.

⸻

1️⃣ Objective

Build an Automated Competitor Data Acquisition Layer that:
	•	Requires only Instagram URL from user
	•	Automatically collects public data
	•	Feeds MIv3 with real signals
	•	Increases confidence without manual filling

No fake inference.
No hallucination.
No silent fabrication.

⸻

2️⃣ System Architecture

A) Competitor Collector Module

Input:
	•	Instagram URL

Output:
	•	competitorId
	•	normalized handle
	•	crawlTarget config
	•	initial profile metadata

Validation:
	•	Must reject invalid URLs
	•	Must deduplicate competitors
	•	Must enforce minimum 3 competitors rule (UI warning if below)

⸻

B) Public Social Scraper (Proxy-based)

For each competitor:

Collect:
	•	Last 30–60 posts/reels
	•	Caption text
	•	Hashtags
	•	Post date
	•	Engagement counts (likes/comments count)
	•	Content type (reel/post/carousel)
	•	Bio text
	•	Link-in-bio

Constraints:
	•	Hard cap on posts collected (configurable)
	•	No unbounded crawling
	•	Token budget awareness

Storage:
	•	Persist raw social signals in dedicated table:
ci_competitor_posts
ci_competitor_metrics_snapshot

No overwriting old data — versioned or timestamped.

⸻

C) Comment Sampling Engine (Controlled)

Not full scraping.

For each competitor:
	•	Sample up to 20 comments per last 10 posts
	•	Extract:
	•	Sentiment drift
	•	Repeated objections
	•	FAQs
	•	CTA triggers

If comments unavailable:
	•	Flag signal coverage
	•	Lower confidence
	•	Never fabricate

⸻

D) CTA Pattern Extractor

Automatically extract:
	•	CTA types (DM, WhatsApp, Call, Link, Offer, Book)
	•	CTA frequency
	•	CTA shift over time

Remove “ctaPatterns missing” permanently once data exists.

⸻

E) Refresh & Scheduling
	•	Default refresh: every 7 days
	•	High volatility: 3 days
	•	Stable: 14 days
	•	72h hard cap enforced
	•	No background silent recompute without logging

Refresh must:
	•	Respect competitorHash
	•	Not create duplicate snapshots
	•	Use existing concurrency lock

⸻

3️⃣ UI Requirements

Competitor Card must show:
	•	Posts collected: X / 30
	•	Comments collected: X / 100
	•	CTA coverage: percentage
	•	Data freshness: days

Add buttons:
	•	“Auto-Fetch Data”
	•	“Refresh Data”

If insufficient data:
Show explicit banner:
“Low confidence due to insufficient competitor signals.”

No generic wording.

⸻

4️⃣ Confidence Model Update

Confidence must now depend on:
	•	Post coverage ratio
	•	Comment coverage ratio
	•	Competitor count (>=3)
	•	Fully qualified competitor count (>=2)

Confidence must not rise artificially if only URLs exist.

⸻

5️⃣ Hard Rules
	•	No LLM scraping.
	•	No synthetic data.
	•	No inferred engagement.
	•	No hallucinated pricing.
	•	Missing data must be surfaced, not hidden.

⸻

6️⃣ Token Safety
	•	Scraper is non-LLM.
	•	MIv3 remains deterministic.
	•	Narrative generation still capped.
	•	No token explosion from competitor count.

⸻

7️⃣ Tests Required

Add automated tests for:
	1.	Competitor with only URL → confidence < 65%
	2.	Competitor with 30 posts → confidence rises
	3.	Competitor with posts but no comments → downgrade but not block
	4.	Duplicate competitorHash → no new snapshot
	5.	Refresh within 72h → blocked
	6.	Proxy blackout → fallback to existing snapshot
	7.	Max competitor count stress test
	8.	3 competitor threshold enforcement

All must pass.
