PHASE 2 — “Light Coverage” Strategy (Reduce Post Requirements Without Killing Quality)
────────────────────────────────
Objective:
Solve the proxy volume problem by reducing required fetch volume while maintaining market usefulness.

Policy Change (recommended for reliability):
- Default analysis window: 12–14 posts from the last 60 days per competitor
- If competitor is low-frequency: expand window to 120 days to reach 12–14 posts
- Refresh cadence: weekly refresh (e.g., every Saturday) OR manual refresh (but one-click orchestration)
- Do NOT crawl years of history by default

Rules:
- If minimum coverage not met → confidence must DOWNGRADE with explicit missing flags (no fabrication)
- Coverage thresholds become “dynamic”:
  - Target: 12–14 posts / 60 days
  - If insufficient → widen to 120 days
  - If still insufficient → mark LOW/UNSTABLE with exact stop reasons

Hard QA Gates:
- First-click job must finish with either:
  - Coverage met, OR
  - Explicit reasons (PRIVATE_ACCOUNT, PROXY_BLOCKED, RATE_LIMITED, NO_MORE_PAGES, etc.)
- Confidence must move logically with coverage (no random 58% while missing core evidence)

Fix Immediately if FAIL:
- Ensure thresholds in UI and engine are aligned
- Ensure cooldown never blocks re-fetch if coverage below target
- Ensure stopReason is always set when fetch is partial
