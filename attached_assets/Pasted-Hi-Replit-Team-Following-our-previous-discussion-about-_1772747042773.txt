# 1. Core Data Collection Fix: Two-Speed Collection

Instagram blocks are not caused by the proxy provider. We already use Bright Data residential proxies. The issue is request behavior (burst scraping, comments pagination, concurrency patterns, and session instability).

To fix this, the collector must operate in two stages.

---

## Fast Pass (Immediate Snapshot)

Fast Pass must complete quickly and provide enough information to generate a valid MI snapshot without triggering Instagram protections.

Fast Pass must collect:

- Competitors: 6–10 accounts
- Posts per competitor: 8–12 posts
- Data per post:
  - caption text
  - hashtags
  - post timestamp
  - engagement counts (likes + comment counts as numbers only)
  - media type if detectable

Fast Pass must NOT collect:

- comment text
- deep pagination
- full history scraping
- heavy endpoints

Fast Pass provides sufficient information to compute:

- similarity diagnosis
- hook and CTA patterns
- angle saturation
- dominance indicators
- preliminary market state

Fast Pass should complete in minutes (target: roughly 3–8 minutes).

---

## Deep Pass (Background Enrichment)

Deep Pass runs automatically after Fast Pass completes and progressively improves the intelligence snapshot.

Deep Pass may:

- increase posts per competitor up to 20–30
- collect deeper signals gradually
- update evidence coverage

Comments must be collected carefully because they trigger most blocks.

Rules for comment collection:

- Only analyze comments from the top 2–3 engagement posts per competitor
- Maximum 30–80 comments per post
- No full comment pagination

This provides enough language data for audience insights without triggering blocks.

---

# 2. Request Pacing and Concurrency Rules

Even residential proxies will fail if request patterns are unnatural.

Implement the following limits:

Global concurrency:
- Maximum 2–4 workers during Fast Pass

Per competitor:
- Only one worker per competitor
- No parallel requests against the same profile

Request pacing:
- One request every 5–12 seconds
- Random jitter between requests

Total request budget must be enforced for each run to prevent burst scraping.

---

# 3. Session Handling (Critical for avoiding blocks)

Instagram blocks often occur because requests appear stateless.

Required behavior:

- Use sticky sessions per competitor for 10–20 minutes
- Maintain consistent:
  - proxy session
  - cookies
  - user agent
  - headers

Do NOT rotate fingerprint or user agent between requests in the same competitor session.

Session identity should remain stable for each competitor scan.

---

# 4. Cache-First Data Collection

The system must avoid re-scraping the same posts repeatedly.

Implement caching rules:

- Store post IDs and metadata locally
- On future runs, collect only new posts
- If competitor data was fetched within the last 12–24 hours, reuse cached data

Comments should never be re-scraped unless explicitly required and within sampling limits.

---

# 5. Safety Mechanisms

### Circuit Breaker

If Instagram responses indicate blocking (login wall, repeated redirects, rate limits):

- immediately stop heavy scraping
- return a partial MI snapshot
- mark the result with reduced confidence
- schedule retry with backoff

### Adaptive Backoff

If response times or block indicators increase:

- automatically reduce concurrency
- increase request delays
- switch temporarily to Fast Pass only

# 6. Self-Updating Market Intelligence (User Experience Requirement)

Market Intelligence must not require manual refresh by the user.

System behavior should be:

1. User runs MI analysis
2. Fast Pass generates the first snapshot
3. Results appear immediately with partial confidence
4. Deep Pass continues enriching data in the background
5. MI snapshot automatically recomputes when new evidence arrives

Users must not be forced to refresh the application.

# 7. Architectural Rule: Engines Produce Signals Only

To avoid long-term conflicts between system modules, enforce the following architecture rule:

Market Intelligence must produce signals and diagnosis only, never strategy decisions.

MI outputs should include:

- evidence signals
- similarity analysis
- hook patterns
- CTA presence
- dominance indicators
- opportunity signals
- confidence metrics

MI must NOT output instructions such as:

- “use this hook”
- “target this audience”
- “run this campaign”

Strategic decisions belong to later engines such as:

- Audience Engine
- Positioning Engine
- Build a Plan
- Storytelling Engines

# 8. Market Activity vs Market Demand (Critical Correction)

Currently MI may interpret low posting frequency or engagement as “low market activity.”

However, this does not necessarily reflect demand.

We must distinguish between:

Market Activity
(content production by competitors)

and

Market Demand
(audience interest and intent)

For example:
Low posting activity may exist in a market with strong audience demand but weak competitors.

MI should therefore output both:

- Market Activity
- Market Demand (or Demand Confidence)

This prevents misinterpretation of opportunities.

---

# 9. Engagement Quality vs Engagement Volume

Engagement count alone is misleading.

MI must distinguish between:

Low-value engagement:
- emojis
- generic praise

High-intent engagement:
- questions
- requests for help
- clarification about methods

Example:

High likes but no real questions → weak intent signal.

Lower likes but many meaningful questions → strong audience demand.

MI should therefore identify:

Audience Intent Signals derived from comment language samples.

Examples:

- fat loss confusion
- beginner workout questions
- diet uncertainty
- program effectiveness concerns

These signals feed Audience and Positioning engines.
