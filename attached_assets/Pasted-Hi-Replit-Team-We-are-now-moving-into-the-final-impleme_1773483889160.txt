Hi Replit Team,

We are now moving into the final implementation phase for MarketMind V1.

This is not a small UI cleanup. This is a full product architecture alignment task.  
All existing engines already exist and already work. The goal now is to unify them into a single, coherent, production-grade operating system with a clean user experience and a strong backend orchestration layer.

Please read this carefully and implement it exactly as specified. I do not want partial interpretation. I want a full, connected, deeply integrated execution.

---

## 1) Core Product Philosophy

MarketMind V1 must behave as an AI Marketing Operating System, not as a collection of disconnected tools or engine screens.

The frontend must feel:
- simple
- guided
- structured
- execution-focused

The backend may be complex, but the frontend must not expose that complexity.

The user should always feel:
- there is one plan
- there is one execution track
- there is one current priority
- the system is telling them what to do next

This means the UI must be reorganized around the plan and execution flow, not around fragmented engine outputs.

---

## 2) Single Strategic Track — Mandatory Rule

The entire product must follow a Single Strategic Track.

That means:
- one active approved plan at a time
- one execution pipeline connected to that plan
- one required work pipeline generated from that plan
- one calendar distribution generated from that plan
- one content creation workload generated from that plan

No parallel disconnected plans.  
No execution outside the active plan.  
No independent content flow that is not traceable back to the approved plan.

All production, scheduling, creation, studio uploads, and publishing states must be traceable back to the current approved plan.

Required system flow:

Engines → Orchestrator → Execution Plan → Calendar / Required Work → Creation → Studio → Publishing

This rule is foundational and must be enforced in data structure, routes, state management, and UI.

---

## 3) Engines Are Not the Frontend Product

All engines remain in the backend logic layer.  
They are not the user-facing product.

The user should not feel they are manually operating 14 engines.

All engine outputs must be absorbed into:
- one execution plan
- one pipeline
- one required work system
- one guided assistant experience

The engines must remain isolated and act like pure computation units.  
They should not directly override each other or write arbitrarily into each other’s domains.

---

## 4) Engine Design Rules — Mandatory

Please enforce these architectural rules:

### 4.1 Engines must behave as pure functions
Each engine receives structured context and returns structured output.

### 4.2 No engine-to-engine direct mutation
Context transfer must happen only through the orchestrator.

### 4.3 AI is for synthesis only
AI should synthesize outputs, explanations, and final plan narrative.  
The engines themselves must remain deterministic and structured.

### 4.4 No decision without market context
No strategic decision may be made without reference to:
- Market Intelligence
- Audience understanding
- business profile
- current plan context
- campaign / execution context where applicable

### 4.5 Complexity must live in the backend
Do not expose orchestration complexity in the frontend.

---

## 5) Engine Priority Matrix — Mandatory Hierarchy

This must be implemented as a fixed architectural rule.

Lower-priority layers cannot override higher-priority layers.

Priority order:

1. Market Reality  
   (Market Intelligence + Audience Engine)

2. Financial Governor / Budget Reality

3. Positioning

4. Offer

5. Messaging / Persuasion

6. Channel

7. Creative

Meaning:

If creative looks strong but financial reality fails, the decision stops.  
If channel is attractive but market reality invalidates it, the decision stops.  
If offer is exciting but positioning is structurally wrong, the downstream layers must not override that.

No engine below may “break” the logic of the engine above it.

This priority matrix must be enforced inside the orchestrator and in final decision synthesis.

---

## 6) AI Orchestrator — Single Entry Execution

We do not want the user manually running engine after engine.

We want one controlled orchestration flow.

Please implement an AI Orchestrator / System Orchestrator that executes the engines sequentially in the correct order through a single top-level action.

The orchestrator must:
- execute engines in order
- persist output after each engine
- pass only relevant context to the next engine
- support checkpointing
- support safe retry from failure point
- prevent prompt overload
- prevent crash from running everything blindly in one unstable burst
- keep the single strategic track intact

Preferred execution order:

1. Market Intelligence  
2. Audience Engine  
3. Positioning  
4. Differentiation  
5. Offer Engine  
6. Funnel Engine  
7. Awareness Engine  
8. Persuasion Engine  
9. Integrity Engine  
10. Statistical Validation  
11. Channel Selection  
12. Budget Governor  
13. Iteration Engine  
14. Retention Engine  
15. BuildUp Plan / Execution Plan synthesis

The user should trigger this as one clean action, not a manual engine-by-engine process.

## 7) BuildUp Plan / Execution Plan — Central Product Layer

The current system has engine outputs, but the actual product must revolve around The Plan.

The plan must not feel like 8 disconnected cards.  
It must feel like a complete strategic and execution blueprint.

The plan must be readable by:
- a simple client
- a founder
- a professional marketer

The plan must transform engine outputs into a coherent operating plan.

Required plan structure:

### 7.1 Strategic Summary
Explain:
- what the strategy is
- who it targets
- what the growth objective is
- why this path was selected

### 7.2 Monthly Objective
Clear business objective:
- leads
- sales
- authority
- retention
- renewal
etc.

### 7.3 KPI Structure
Explicit KPIs with targets:
- primary KPI
- secondary KPI
- performance expectations
- cadence expectations

### 7.4 Content Distribution
What content types are required and why:
- reels
- posts
- stories
- carousels
- supporting assets

### 7.5 Creative Testing
What will be tested:
- hooks
- offers
- formats
- creatives
- narratives
- CTA styles

### 7.6 Budget Allocation
How budget is distributed and why.

### 7.7 KPI Monitoring
What metrics matter during execution.

### 7.8 Competitive Watch
What market or competitor movement the system is watching.

### 7.9 Risk Triggers
What conditions will trigger caution, hold, review, or correction.

This plan must then become the source for Calendar and Required Work.

## 8) Required Work System — Mandatory

This is critical.

The plan must automatically generate Required Work Units.

Example:
- 5 reels
- 10 posts
- 30 stories
- etc.

These units must drive execution.

Required Work states must be clearly tracked.

Minimum status model:
- Required
- Fulfilled
- Remaining
- Generated
- Scheduled
- Published
- Failed

The user should always know:
- what is required
- what is already done
- what is still missing
- what is today’s work

When content is created and sent to Studio, Required Work must automatically decrement.

This must not be manual.

---

## 9) Calendar Integration — Mandatory

The plan must automatically generate calendar entries.

Plan → Calendar distribution

Calendar entries must contain:
- content type
- assigned day / date
- relation to plan
- status
- lifecycle stage

Calendar must not be isolated from the plan.

The current “Open Calendar” direction is correct, but it must be deeply connected:
- approved plan generates calendar
- calendar reflects required work
- calendar completion updates progress
- created assets map back to scheduled items

---

## 10) Creation Integration — Mandatory

The Creation section must read directly from Required Work / Calendar state.

The user should not enter Creation and face an abstract empty generator.

Creation must show:
- what is required today
- what is required this week
- what is still missing from the current plan

Examples:
- 1 reel required today
- 2 posts due this week
- 3 stories remaining

Creation should allow:
- idea generation
- brief dispatch
- content generation
- sending result to Studio

Creation must be execution-aware, not isolated.

---

## 11) Studio Integration — Mandatory

Studio must become the asset repository and review layer.

Minimum asset states:
- Draft
- Generated
- Ready
- Approved
- Published

When the user creates content or uploads generated content:
- it goes to Studio
- it connects to the relevant required work item
- it updates plan execution state

If Meta API is not connected:
- publishing remains manual
- but state transitions must still be supported

If Meta API becomes available later:
- Studio should be able to push to autopublishing

---

## 12) Publishing Logic

### 12.1 Manual publishing is primary fallback

### 12.2 Studio must still be publish-aware
Even if publishing is manual, the asset lifecycle must support:
- ready to publish
- published manually

- plan progress updated

Do not hard-wire the system to Meta dependency.

## 13) AI Agent — Role Definition (Critical)

This must be implemented very carefully.

The MarketMind Agent must behave as a strategic operations manager, not as a replacement for the engine system.

The agent is not the strategy engine.  
The agent is not the source of truth.  
The agent is not allowed to invent disconnected strategy outside the approved plan.

The agent’s job is to:
- explain the system
- interpret the plan
- guide the user
- suggest next actions
- help the user understand what to create
- help the user ask better questions
- trigger system actions
- dispatch work into Creation when needed

The agent must feel like:
a manager / guide / operator

Not:
a random answer machine

### Agent can:
- read system state
- explain engine outputs
- explain the plan
- suggest ideas
- trigger orchestrator
- trigger creation briefs
- help the user understand priorities

### Agent cannot:
- replace engines
- rewrite final strategy arbitrarily
- override financial governor
- override market reality
- create disconnected strategy outside current plan
- mutate engine outputs directly

This point is non-negotiable.

## 14) Multi-Chat Architecture for the Agent — Mandatory

We need free-form multiple chats, not a single giant thread.

However, the agent must not depend on long chat history to understand the system.

### Required behavior:
- user can open New Chat at any time
- new chat must still understand the system immediately
- old chats remain visible in history
- conversation continuity is optional, not required for system understanding

### Critical architectural rule:
Agent understanding must come from System Context, not chat history.

This means every new chat must load:
- business profile
- current campaign
- approved plan
- latest execution status
- required work state
- calendar status
- studio status
- relevant engine outputs
- warnings / blockers / validation state

Chat history should exist only for:
- user review
- conversation continuity
- recalling prior discussion in that thread

Do not build the agent so that a new chat becomes “dumb” unless it rereads the old conversation.

That is the wrong architecture.

We want:
- free-form chats
- clean reset when needed
- system-aware new conversations
- fast performance
- scalable memory design
