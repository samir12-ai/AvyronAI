## 1) Global Reliability Hardening Before and During Awareness Engine

Before treating any downstream reasoning as trustworthy, the system must protect itself against weak input data and structural drift.

Please implement or finalize the following global safeguards:

### A. Data Reliability Guard
We need a unified reliability layer so downstream engines do not build strong strategic conclusions on top of weak data.

This guard must evaluate:

- signal density  
- signal diversity  
- narrative stability  
- competitor validity  
- market maturity  

If reliability is weak, the system must:

- downgrade confidence
- attach reliability diagnostics
- avoid overconfident strategic outputs

This should work in coordination with existing protections such as:

- Sample Bias Detection
- Real Data Ratio Guard
- Authority Weighting

### B. Confidence Normalization
Different engines currently produce confidence levels that are not fully comparable.

Implement normalized confidence handling so the system evaluates cross-engine confidence consistently instead of comparing unrelated raw values.

### C. Snapshot Stability Policy
To prevent long-term database inflation and messy state growth:

- define a snapshot pruning or archive policy
- preserve upstream traceability
- keep accountId, campaignId, engine version, timestamps, and snapshot references intact

### D. Engine Compatibility Matrix
Prevent version drift across engines.

If one engine is running on incompatible upstream versions, the system must detect this and downgrade or warn rather than silently proceeding.

### E. Revalidation Loop Guard
Prevent repeated circular revalidation chains where one update keeps triggering another indefinitely.

A validation session or equivalent protection should prevent endless loop behavior across Offer, Funnel, Integrity, and Awareness.

### F. Route and Runtime Verification
As always, please avoid the common failure where code is updated but the live system continues running an older build.

After implementation:

- rebuild affected modules
- restart backend/runtime
- verify engine version
- verify routes live in the running environment
- verify real snapshot chain behavior

Do not close the task unless the live environment is confirmed to be using the new logic.

## 4) How Awareness Engine V3 Must Be Built

Build the engine as a structured decision engine with these layers:

### Layer 1 — Market Entry Detection
Determine the best entry route into the market.

Possible entry routes may include:

- pain entry
- opportunity entry
- myth-breaker entry
- authority entry
- proof-led entry
- diagnostic entry

The choice must be based on:

- audience readiness
- trust level
- problem intensity
- market maturity
- competitor pressure

### Layer 2 — Awareness Readiness Mapping
Classify the audience awareness stage:

- unaware
- problem aware
- solution aware
- product aware
- most aware

The awareness route must match this stage correctly.

### Layer 3 — Attention Trigger Mapping
Identify the correct trigger classes such as:

- hidden cost trigger
- missed opportunity trigger
- outdated method trigger
- authority gap trigger
- trust breakdown trigger
- competitor weakness trigger

These are trigger categories only, not final messaging copy.

### Layer 4 — Narrative Entry Alignment
Ensure awareness logic aligns with:

- positioning narrative
- positioning enemy
- differentiation pillars
- offer mechanism
- funnel structure

If awareness entry contradicts positioning or differentiation, confidence must be reduced.

### Layer 5 — Awareness-to-Funnel Fit
Verify that the awareness route logically feeds the selected funnel.

Examples:

- challenge funnels need curiosity or diagnostic entry
- consultation funnels require trust or authority entry
- application funnels require pre-qualified awareness

If the awareness route does not logically feed the funnel, the system must downgrade the result.

### Layer 6 — Trust Readiness Guard
If the market requires early trust-building, the awareness path must be:

- proof-led
- authority-led
- transparency-led

It must not use hype-first awareness when trust is the real barrier.

### Layer 7 — Generic Awareness Detector
Detect and penalize weak generic awareness logic such as:

- grow your business
- scale faster
- improve marketing

If the awareness route is generic and market-noise-like, apply a score penalty.

### Layer 8 — Awareness Strength Scoring
Score the awareness architecture using dimensions such as:

- audience readiness fit
- positioning alignment
- funnel compatibility
- trust suitability
- distinctiveness
- entry clarity
- friction reduction
- practical deployability
