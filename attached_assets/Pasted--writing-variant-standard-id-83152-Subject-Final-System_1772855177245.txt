Subject: Final System Hardening — Zero-Failure Guarantee for MI → Audience → Positioning Pipeline

Hello Replit,

We want to close the Market Intelligence investigation and ensure the system reaches full reliability (10/10) with no possibility of false blocks, stale miscalculations, or UI/backend inconsistencies.

The previous fixes addressed two critical issues:

• Freshness calculation using the oldest post instead of the newest
• Frontend misreading the MI snapshot structure (snapshot.id nesting)

Both fixes were correct. Now we want to perform final hardening to guarantee that this class of issue can never occur again.

Please implement the following system safeguards.

⸻

	1.	Guaranteed Correct Freshness Calculation

Ensure the freshness computation always uses the newest available signal timestamp.

Freshness must be calculated using the most recent timestamp among:

• competitor posts
• competitor comments
• competitor metrics snapshots

Historical posts must never mark the dataset as stale.

Add a validation rule:

If calculated freshness > 365 days while recent signals exist → flag calculation anomaly and recompute.

This prevents future regression of the 744-day error.

⸻

	2.	Snapshot Response Normalization Layer

All frontend components must consume normalized snapshot objects.

Implement a normalization adapter:

normalizeEngineSnapshot(response)

Input example:

{ snapshot: {…}, output: {…} }

Normalized structure returned to UI:

{
snapshot,
output,
metadata
}

No UI component should access raw API responses directly.

Apply this normalization to:

• MI snapshot endpoint
• Audience snapshot endpoint
• Positioning snapshot endpoint

This prevents future nested response bugs.

⸻

	3.	Engine Readiness Validation

Replace simple checks such as:

hasMI = !!miSnapshot?.id

with strict readiness verification.

Create helper:

getEngineReadinessState()

MI is READY only if:

• snapshot exists
• snapshot.id valid
• snapshot.campaignId matches current campaign
• freshness within allowed threshold
• snapshot integrity verified

Audience and Positioning must depend on this readiness helper.

⸻

	4.	Snapshot Integrity Verification

Before a snapshot is accepted as valid, verify:

• snapshot.id exists and is a valid UUID
• snapshot.createdAt valid
• snapshot.campaignId matches current campaign
• snapshot.engine_version matches expected engine version
• output payload exists and is non-empty

Invalid snapshots must trigger a refresh instead of being used.

⸻

	5.	Automatic Refresh on Stale Data

If MI data becomes stale:
	1.	Detect stale snapshot
	2.	Trigger automatic refresh
	3.	Set engine state to REFRESHING
	4.	Generate new snapshot after refresh

The system must never permanently block analysis when refresh is possible.

⸻

	6.	Scraping Block Resilience

If scraping for a competitor fails due to platform blocking:

• mark competitor temporarily unavailable
• continue analysis using remaining competitors if minimum threshold is met
• schedule retry later

The entire pipeline must never halt because one competitor failed to fetch.

⸻

	7.	Engine State Machine

Explicitly define MI states:

READY
REFRESH_REQUIRED
REFRESHING
REFRESH_FAILED
BLOCKED

UI must reflect these states accurately.

BLOCKED should only occur if:

• dataset integrity fails
• refresh repeatedly fails
• minimum competitor threshold cannot be met

⸻

	8.	Dependency Enforcement

Audience and Positioning must only run if MI state = READY.

Verify:

• MI snapshot exists
• snapshot freshness valid
• snapshot campaignId matches
• snapshot integrity confirmed

If MI is refreshing, downstream engines must wait.

⸻

	9.	Defensive Diagnostics

Add structured diagnostic logs showing engine state resolution.

Example:

MI Status:
snapshotFound: true
freshnessDays: 4
freshnessValid: true
engineState: READY

This ensures future issues can be diagnosed instantly.

⸻

	10.	Hard Verification Tests

Run validation tests covering:

• freshness calculation edge cases
• nested snapshot responses
• stale snapshot handling
• automatic refresh triggering
• scraping block scenarios
• competitor partial failures
• dependency enforcement for Audience and Positioning

The system must confirm that valid data never appears blocked and stale data always triggers refresh first.

⸻

Final Objective

The MI → Audience → Positioning pipeline must become:

• deterministic
• self-recovering
• resistant to stale data
• immune to UI/backend mismatches

Please confirm once these safeguards are implemented and verified.

Thank you.
